# Role: Architect

> **Headcount:** 1 person | **Typical title:** Principal Engineer, Staff Engineer, or Solutions Architect | **Autonomy tier scope:** Owns all HITL gates for ADR approval, `design.md` approval, and architectural integrity enforcement

---

## Role Purpose

The Architect is the **guardian of long-term technical integrity** in an AI-native pod. Where agents optimise for the local task — writing code that compiles, tests that pass, specs that are internally consistent — the Architect optimises for the global system: cohesion across services, fitness for the target infrastructure, and the sustainability of decisions over years, not sprints.

In an AI-native workflow, the Architect's job changes meaningfully from a traditional role:
- **Less time** designing solutions from scratch (agents draft designs)
- **More time** critically reading agent-generated designs and ADRs for hidden assumptions, context blindspots, and architectural anti-patterns
- **New responsibility:** Ensuring that ADRs are injected into agent context correctly so that agents do not contradict approved decisions

The Architect is not a bottleneck; they are a **quality gate with context**. Every ADR approval is a deliberate act, not a formality.

## Day-to-Day Responsibilities

### Weekly Rhythm

**Monday — ADR backlog review:**
- Review any ADR drafts generated by the Design Agent during the previous sprint
- Assess each draft: approve, reject with rationale, or return for revision
- Check the ADR injection manifest (which ADRs are being loaded into which agent contexts) for completeness

**Daily — Design review:**
- Review open `design.md` documents awaiting approval (target: <24h turnaround for standard designs)
- Flag any architectural concerns in the design before the Implementation Agent begins work
- Scan PR reviews from the Review Agent for architectural drift signals

**Wednesday — Standards maintenance:**
- Check whether new Spring Boot dependencies introduced in PRs are consistent with `framework/standards/runtime-standards.md`
- Review any agent-generated OpenAPI specs for contract hygiene violations (missing error schemas, inconsistent naming, undocumented headers)
- Confirm that `distroless` base images and multi-stage Dockerfiles are used in all new container definitions

**Friday — Metrics review:**
- Review ADR compliance rate from the audit log
- Check architectural drift indicators (see metrics section below)
- Update `CLAUDE.md` architecture section if any new patterns have been standardised

## Java/Spring Boot Specific Responsibilities

This section applies when the product runtime is Java 21 + Spring Boot 3.x (the default). For other runtimes, consult the relevant section in `framework/standards/runtime-standards.md`.

### Spring Patterns Review

The Architect reviews agent-generated Java code for compliance with the following patterns:

**Dependency Injection:**
- Constructor injection required; field injection (`@Autowired` on fields) is prohibited
- `@Qualifier` usage must be justified — prefer type-specific beans where possible
- Circular dependencies must be resolved at design time (not via `@Lazy` workarounds)

**Layering:**
- Controller → Service → Repository is the required layering; no business logic in controllers
- `@Transactional` belongs on Service layer, never on Controller or Repository
- Domain models must not leak into Controller responses; use dedicated DTOs/response records

**Spring Data JPA:**
- Named queries preferred over JPQL strings in annotations for complex queries
- `@Query` with native SQL requires an ADR justifying the departure from JPQL
- Projection interfaces for read-only queries; no entity fetching for read-only flows
- N+1 detection: the Architect must flag any relationship without explicit fetch strategy

**Spring MVC / WebFlux:**
- REST controllers use `@RestController` and follow standard HTTP semantics (per ADR-API-001 or equivalent)
- Reactive (`WebFlux`) stack must be declared in `proposal.md` before use; cannot be introduced by an agent without an ADR

**Configuration:**
- `application.yml` only; no `application.properties`
- Environment-specific overrides via Spring Profiles; no hardcoded environment strings
- All secrets via Spring Cloud Vault or GCP Secret Manager — never in config files or env vars in container spec

### Dependency Injection Hygiene

The Architect reviews `pom.xml` (or `build.gradle`) changes for:
- No snapshot or pre-release dependencies in production code
- No duplicate transitive dependencies at conflicting versions (run `mvn dependency:tree` check in CI)
- Spring Boot parent BOM version pinned; no individual Spring library versions overriding BOM
- New dependencies require ADR if they introduce a new architectural pattern (e.g. first use of a messaging library, first use of a cache abstraction)

### API Contract Hygiene

The Architect validates OpenAPI 3.1 contracts generated by agents:

**Naming:**
- Paths: lowercase kebab-case (`/product-variants`, not `/productVariants`)
- Query parameters: camelCase
- JSON fields: camelCase
- HTTP methods correctly applied (GET/POST/PUT/PATCH/DELETE per ADR-API-001)

**Schema:**
- All request bodies have `required` fields declared
- All response schemas documented, including 4xx and 5xx error shapes
- `$ref` used for reusable schemas; no inline duplication
- Versioning: `v1` in path prefix by default; version bump requires Orchestrator approval

**Validation:**
- `springdoc-openapi` generation is the source of truth; no handwritten OpenAPI YAML
- Contract tests (Spring Cloud Contract or Pact) must cover all consumer-facing endpoints

### Data Model Integrity

The Architect reviews Flyway migration scripts generated by agents:

- Migrations are additive only (no column drops, no table drops) unless explicitly approved via HITL gate
- All foreign keys declared; no implicit relationships
- Index strategy justified in ADR for any table expected to exceed 100k rows
- Soft delete pattern (via `deleted_at` column) required; hard deletes require HITL approval

## Key Skills Required

### Architecture Patterns
- Microservices decomposition (bounded context identification, seam detection)
- Event-driven architecture (Kafka, Spring Cloud Stream)
- API gateway patterns (rate limiting, auth, routing)
- Data consistency patterns (sagas, outbox pattern, eventual consistency)
- Caching strategies (L1/L2 cache, cache invalidation patterns)

### ADR Writing
ADRs in this framework follow the MADR (Markdown Architectural Decision Records) format:

```markdown
# ADR-NNN: [Short Title]

## Status
[Proposed | Accepted | Deprecated | Superseded by ADR-NNN]

## Context
[What is the problem? What forces are at play?]

## Decision
[What was decided?]

## Consequences
[What are the positive, negative, and neutral consequences?]

## Alternatives Considered
[What other options were evaluated and why were they rejected?]
```

The Architect owns the ADR backlog: numbering, status management, and injection into agent context.

### Reading Agent-Generated Output Critically

The Architect must resist the authority illusion of agent output. Key heuristics:
- **"Does this design acknowledge the constraints in the ADRs?"** — Agents that have not been given ADRs in context will generate designs that contradict them.
- **"What did the agent not mention?"** — Absent concerns (performance, failure modes, operational burden) are often more informative than present ones.
- **"Would I have to change this design in 6 months?"** — Agents optimise for the stated requirement; the Architect optimises for the unstated future.

## HITL Gates Owned by the Architect

| Gate | Trigger | What Must Happen | SLA |
|---|---|---|---|
| **ADR Approval** | Any ADR generated by the Design Agent or proposed by a team member | Architect reads, challenges assumptions, approves or rejects with written rationale | 48h from submission |
| **`design.md` Approval** | Design Agent generates `design.md` | Architect reviews against ADRs and runtime standards; may request revision | 24h from submission |
| **New Dependency ADR** | Any PR introducing a new architectural dependency | Architect reviews ADR for new dependency; approves or rejects | 24h from submission |
| **Schema Migration Review** | Any Flyway migration that drops or alters existing columns | Architect reviews impact; must co-approve with Ops | Before CI merge |
| **Runtime Standards Change** | Any proposed change to `framework/standards/runtime-standards.md` | Architect reviews, writes supporting ADR, approves | 48h from proposal |
| **OpenAPI Contract Approval** | New API version or breaking change detected by contract linter | Architect approves contract change | 24h from detection |
| **ADR Override** | Agent or team member proposes departing from an accepted ADR | Architect reviews, may open "Supersedes" ADR, must approve before departure | Synchronous if possible |

## How Architects Inject ADRs into Agent Context

ADRs are the primary mechanism for ensuring agents make decisions consistent with approved architecture. The injection process:

### Step 1 — ADR Manifest
Each product maintains an ADR manifest at `products/<product>/adr/manifest.json`:

```json
{
  "inject_always": ["ADR-001", "ADR-002", "ADR-003"],
  "inject_by_phase": {
    "implementation": ["ADR-010", "ADR-011"],
    "testing": ["ADR-020"],
    "deployment": ["ADR-030", "ADR-031"]
  },
  "inject_by_service_type": {
    "rest-api": ["ADR-API-001", "ADR-API-002"],
    "data-pipeline": ["ADR-DATA-001"]
  }
}
```

### Step 2 — Context Package Assembly
When an agent starts a task, the context package is assembled automatically:
1. `products/<product>/CLAUDE.md` (always)
2. ADRs from `inject_always` list (always)
3. ADRs from `inject_by_phase` for the current phase
4. ADRs from `inject_by_service_type` for the product's service type
5. Active `spec.md` section(s) relevant to the task
6. `glossary.md`

The Architect is responsible for maintaining the manifest and ensuring newly approved ADRs are added to the appropriate injection lists.

### Step 3 — CI Enforcement
A CI step (Harness pipeline stage: `architectural-lint`) runs on every PR to verify:
- No approved ADR is contradicted in the changed files
- New patterns introduced must reference an ADR (checked via code comment convention: `// ADR-NNN`)
- OpenAPI contract diffs are validated against the contract hygiene checklist

## Metrics the Architect Tracks

| Metric | Target | Measurement Method | Review Cadence |
|---|---|---|---|
| ADR Compliance Rate | >95% of PRs pass architectural lint without ADR violations | CI lint pass rate in Harness | Weekly |
| ADR Backlog Age | No ADR draft older than 5 business days without decision | ADR manifest timestamp audit | Weekly |
| Architectural Drift Index | <3 unchecked drift indicators per sprint | Manual review of PR patterns | Bi-weekly |
| Design Approval Turnaround | Median <24h, P95 <48h | Phase timestamp audit | Monthly |
| New Dependency Rate | Track trend; flag if >2 new architectural dependencies per sprint | `pom.xml` diff analysis | Per sprint |
| Contract Break Rate | 0 unannounced breaking changes in production API | Contract test failure rate | Per deployment |

### Architectural Drift Indicators
These are signals the Architect watches for in agent-generated PRs:
- Business logic appearing in `@Controller` or `@RestController` classes
- `@Autowired` field injection
- Native SQL queries without an ADR
- Logging sensitive data (PII, credentials) in log statements
- Missing `@Valid` on request body parameters
- Hardcoded environment strings or credentials
- Spring `@Profile` annotations on business logic classes (not just config)
- Missing health check endpoints in new services

## Alternative Agent Adapters Note

When the Architect updates ADR injection configuration, the same ADR manifest drives context injection regardless of the agent backend in use:
- **Claude Code:** ADRs are listed in the CLAUDE.md `@file` section or slash command context
- **LangGraph:** ADR content is loaded as a document node in the graph context state
- **Google ADK:** ADRs are loaded as tool-accessible knowledge base entries
- **CrewAI:** ADRs are added to the agent's `knowledge_sources` list

The Architect does not need to understand each adapter's internals — the ADR manifest is the single source of truth.
