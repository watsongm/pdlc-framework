# Role: Orchestrator

> **Headcount:** 1 person | **Typical title:** Product Lead, Tech Lead, or Product Engineer | **Autonomy tier scope:** All tiers — owns HITL gates for spec and agent team direction

---

## Role Purpose

The Orchestrator is the **translation layer between human product intent and machine-executable work**. In an AI-native pod, this role is not about writing feature tickets — it is about writing *context* that agents can act on reliably, and *governing* the agent team so that what gets built matches what was intended.

The Orchestrator answers two questions continuously:
1. **"Are we building the right thing?"** — spec quality, stakeholder alignment, horizon positioning
2. **"Is the agent team building it correctly?"** — context integrity, escalation response, ADR compliance

Without a strong Orchestrator, agent teams produce a flood of plausible-looking output that diverges from intent within days. The Orchestrator is the single point of accountability for preventing that drift.

### What the Orchestrator Does vs. What Agents Do

| Orchestrator (Human) | Agents |
|---|---|
| Writes `proposal.md` from stakeholder conversations | Generate `spec.md` from `proposal.md` |
| Decides which horizon (H1/H2/H3) an initiative belongs to | Tag tasks with horizon labels (HOOTL) |
| Approves final `spec.md` before design phase begins | Generate `design.md` and initial ADRs |
| Resolves conflicting stakeholder priorities | Surface conflicts as escalations with options |
| Owns `CLAUDE.md` structure and freshness | Read `CLAUDE.md` as primary context source |
| Decides when to override an ADR | Flag ADR conflicts and escalate |
| Signs off on agent team composition changes | N/A |
| Reviews escalation messages and unblocks agents | Generate escalation messages with decision options |
| Runs monthly governance reviews | Produce audit log summaries (HOOTL) |
| Manages relationships with stakeholders outside the pod | Gather structured information for stakeholder comms |

## Day-to-Day Responsibilities

### Weekly Rhythm

**Monday — Sprint kick-off:**
- Review `tasks.md` generated by the Task Decomposition Agent
- Confirm priority ordering against current stakeholder commitments
- Verify all active tasks have sufficient context in `CLAUDE.md` and linked spec sections
- Unblock any pending escalations from the previous sprint

**Daily — Escalation response:**
- Review any escalation messages from agents (target response SLA: 2 hours for P2, same-day for P3)
- Approve or provide clarifying context
- Update `CLAUDE.md` if the escalation revealed a context gap

**Wednesday — Mid-sprint check:**
- Review agent PR descriptions for spec alignment (HOTL)
- Check that the Architect has reviewed any ADR-touching PRs
- Verify cost-per-feature is tracking within budget

**Friday — Retrospective input:**
- Note context quality issues encountered during the week
- Update `glossary.md` with any new domain terms that appeared in specs or PRs
- Confirm `proposal.md` for next sprint's features is ready for spec agent input

### Ongoing Responsibilities

- **Context Engineering:** Keep `CLAUDE.md` accurate, comprehensive, and under 8,000 tokens (enforced). This is a first-class engineering responsibility, not a documentation task.
- **Stakeholder Communication:** Translate agent-produced specs and design documents into human-readable summaries for non-technical stakeholders.
- **Horizon Governance:** Review the horizon assignment of each active initiative monthly. Reclassify as appropriate.
- **Agent Team Health:** Monitor aggregate metrics. Escalate framework-level issues to the Architect.

## Key Skills Required

### Spec Writing
The Orchestrator must be able to write `proposal.md` entries that are:
- **Precise enough** for the Spec Agent to generate a testable spec without guessing
- **Bounded enough** to fit within a single sprint's scope
- **Contextually rich** — includes domain constraints, non-functional requirements, and known risks upfront

Bad `proposal.md` produces bad `spec.md`, which produces bad tasks, which produces bad code. The Orchestrator is the root cause of most agent team failures.

### Context Engineering
Context engineering is the discipline of curating what agents see when they start work. The Orchestrator must understand:
- What the `CLAUDE.md` pattern is and how agents consume it
- How to inject ADRs, glossary, and spec references into agent context without exceeding token limits
- How to structure instructions so that agent behaviour is predictable and auditable

See `framework/standards/agent-interface.md` for the context injection standard.

### Stakeholder Communication
The Orchestrator must translate in both directions:
- **Inward:** Turn stakeholder language ("we need better search") into spec language ("implement fuzzy text search over `Product.description` using PostgreSQL full-text search with `tsvector`; response time P99 <200ms")
- **Outward:** Turn spec and design documents into executive summaries, demo narratives, and risk communications

### Decision Framing
When agents escalate decisions, the Orchestrator must be able to:
- Evaluate options quickly with incomplete information
- Distinguish between "agent needs more context" (provide it) and "agent has found a genuine ambiguity" (resolve it via ADR or spec update)
- Avoid the rubber-stamp anti-pattern (see below)

## Decision Authority

The Orchestrator has final sign-off authority on:

| Decision | Notes |
|---|---|
| Approval of `proposal.md` before it enters the spec pipeline | Must be written by the Orchestrator; agents cannot write proposals |
| Approval of `spec.md` | After Spec Agent generation; before design phase |
| Priority ordering of `tasks.md` | After Task Decomposition Agent generation |
| Agent team composition changes (adding/removing agents) | Requires framework ADR update |
| CLAUDE.md structural changes | Notify Architect; no blocking approval needed |
| Horizon tier assignments | Monthly review |
| Escalation responses for spec-related questions | Within SLA |
| Go/no-go for entering the next phase | Each phase gate |

The Orchestrator does **not** have unilateral authority over:
- ADR content or approvals (Architect)
- Production deployments (Ops + HITL gate)
- Architecture decisions in `design.md` (Architect)
- Test coverage requirements (Engineer)

## HITL Gates Owned by the Orchestrator

| Gate | Trigger | What Must Happen |
|---|---|---|
| **Proposal Approval** | Before Spec Agent runs | Orchestrator writes and self-approves `proposal.md`; no agent involvement |
| **Spec Approval** | After Spec Agent generates `spec.md` | Orchestrator reviews spec for intent accuracy; Architect reviews for feasibility; both must approve |
| **Task Priority Sign-off** | After Task Decomposition Agent generates `tasks.md` | Orchestrator confirms priority order; no blocking required for task content |
| **Phase Gate — Discovery to Spec** | After discovery report produced | Orchestrator reviews discovery findings; confirms scope before spec pipeline begins |
| **Phase Gate — Spec to Implementation** | After `design.md` and all ADRs approved | Orchestrator gives final go-ahead to begin implementation |
| **Escalation Response** | Any agent escalation tagged `ORCHESTRATOR` | Must respond within SLA (P1: 30min, P2: 2h, P3: EOD) |

## Interaction Patterns with Each Agent Team

### Spec Agent
- **Input from Orchestrator:** `proposal.md`, `CLAUDE.md`, `glossary.md`
- **Output to Orchestrator:** `spec.md` (draft, pending approval)
- **Typical interaction:** Orchestrator reviews spec, identifies missing non-functional requirements or wrong assumptions, adds corrections to `proposal.md` and re-runs the agent (not by editing `spec.md` directly)

### Discovery Agent
- **Input from Orchestrator:** Codebase access, existing documentation links, key people to interview (as structured notes)
- **Output to Orchestrator:** `discovery-report.md`, `risk-register.md`, `inferred-spec.md`
- **Typical interaction:** Orchestrator validates the inferred spec against actual product intent; often the most gap-revealing interaction

### Task Decomposition Agent
- **Input from Orchestrator:** Approved `spec.md` + `design.md`
- **Output to Orchestrator:** `tasks.md` with sprint-sized tasks
- **Typical interaction:** Orchestrator reorders tasks by business priority (agent orders by technical dependency)

### Implementation Agent
- **Input from Orchestrator:** `CLAUDE.md` (indirectly, as part of agent context)
- **Typical interaction:** Mostly HOTL; Orchestrator only directly intervenes on spec-conflict escalations

### Review Agent
- **Output to Orchestrator:** PR descriptions with spec compliance notes
- **Typical interaction:** Orchestrator may read review summaries; Engineer handles detailed review

## Metrics the Orchestrator Tracks

| Metric | Target | Source |
|---|---|---|
| Spec Coverage | >90% of tasks traceable to a spec section | Audit log + task tracker |
| Spec Rework Ratio | <15% of specs require major revision after agent run | Git history of `spec.md` |
| Escalation Response Time | P2 <2h, P3 <EOD | Escalation log |
| Proposal-to-Spec cycle time | <1 day for routine features | Phase timestamps |
| CLAUDE.md freshness | No section >30 days stale | File modification timestamps |
| Horizon Distribution | Target: >60% H1, <30% H2, <10% H3 per sprint | Quarterly review |

## Anti-Patterns to Avoid

### The Rubber Stamp
**What it looks like:** Orchestrator approves `spec.md` within 30 seconds of agent generation without reading it. The spec goes into design and implementation phases with subtle errors that compound into significant rework.

**Why it happens:** Agent output looks authoritative. It is well-structured, verbose, and confident. Approving without reading feels like trusting the agent.

**Prevention:**
- Set a personal rule: minimum 15 minutes for spec review regardless of length
- Use the spec validation checklist in `phases/02-spec/README.md`
- Have the Spec Agent also generate a "what I assumed" section in `spec.md` — review that list critically

### Over-Delegating Context Engineering to Agents
**What it looks like:** Orchestrator lets agents update `CLAUDE.md` based on what they learn during tasks. The file gradually drifts into a collection of agent observations rather than a curated human-intent document.

**Why it happens:** Agents are good at summarising context they encounter. It feels efficient.

**Prevention:**
- Agents may *suggest* `CLAUDE.md` updates via an escalation; they must not write to it directly
- Orchestrator reviews and applies any suggested updates deliberately
- Treat `CLAUDE.md` like production configuration: reviewed, versioned, intentional

### Spec Inflation
**What it looks like:** `proposal.md` grows to describe six months of work. The Spec Agent produces a 40-page `spec.md`. Tasks are enormous. Nothing ships.

**Why it happens:** Orchestrators who come from waterfall planning backgrounds try to specify everything upfront.

**Prevention:**
- Hard limit: one `proposal.md` per sprint-sized initiative
- If a proposal is growing, it needs to be split into multiple proposals with a sequencing decision
- Use the "what would we ship in week 1?" question to scope proposals

### Treating Agent Output as Ground Truth
**What it looks like:** An Implementation Agent generates a solution that technically compiles and passes tests but misses the actual user need. No one catches it because the spec was ambiguous.

**Why it happens:** Confirmation bias — the output looks complete and correct.

**Prevention:**
- Orchestrator attends (or reviews recordings of) demos of agent-built features
- Maintain a direct link between `spec.md` acceptance criteria and the actual user story in stakeholder language
- Post-delivery: Orchestrator confirms with stakeholders that the feature solved the problem
